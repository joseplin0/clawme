# UI 实现规则

这份文档约束的是当前项目里的日常实现，不是品牌愿景，也不是灵感参考。
如果长期设计稿与这里冲突，以这里和当前代码约定为准。

## 目标

- 保持界面轻一点，不靠大量描边堆层级
- 优先用留白、排版、背景层次和阴影建立结构
- 避免不同页面因为个人习惯出现过重的“框感”

## 边框规则

### 默认原则

- 默认不要加边框。
- 只有当边框承担了明确的信息分组、分隔或可交互暗示作用时，才允许添加。
- 如果去掉边框后信息层级仍然清楚，就不应该保留边框。

### 允许使用边框的场景

- 页面骨架分隔：如侧边栏分界、顶部/底部导航分界
- 列表或表单中的结构分隔：如分组标题下的分隔线、输入框焦点边界
- 需要弱轮廓的浮层或卡片：在纯白背景上仅靠阴影无法稳定识别时
- 状态表达：选中、错误、聚焦、拖拽目标等需要明确反馈时

### 不建议使用边框的场景

- 只是为了“看起来像一个模块”就给整块内容加框
- 卡片已经有足够留白、背景差异或阴影时，再额外套一层边框
- 在同一个区域里同时使用明显背景差、明显阴影和明显边框，造成重复表达
- 给菜单、空状态、说明区默认套边框，除非它确实需要和背景分开

## 替代手段优先级

当需要建立层级时，优先按下面顺序考虑：

1. 留白和间距
2. 字号、字重和文本层级
3. 背景色或透明度变化
4. 阴影或模糊
5. 最后才是细边框

## 实现细则

- 能用 `border-b` / `border-t` 做分隔时，不要默认给整个容器一圈 `border`
- 能用 `shadow` 或 `bg-elevated` 解决的浮层，不要先上整圈描边
- 边框必须是弱边框，优先使用 `border-default/30` 到 `border-default/60` 这类低对比度值
- 不要为了“保险”同时写 `border` 和很重的 `shadow`
- 新页面或新模块提交前，自查一次：这个边框删掉后，结构是否仍然成立

## 对当前项目的直接约束

- 新增页面、弹层、菜单、卡片时，不要默认外包一层边框
- 如果沿用现有组件样式中的边框，需要先判断它是不是结构必需，而不是机械继承
- 以后我在实现里默认按这份规则执行；如果某个场景确实需要更强边框，会明确说明原因
